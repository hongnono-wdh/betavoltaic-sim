# 方案 B 设计 · 技能现场生成 C++(以方案 A 为基准)

- **日期**:2026-06-06
- **前提**:方案 A(参数化引擎 `betasim` + 技能)已交付并验证(见验证报告)。
- **目标**:让技能在面对**超出方案 A 参数范围**的需求时(全新几何/打分/物理),**现场生成一份新的 Geant4 C++ 程序**;并用方案 A 当"标准答案"自动检验生成质量,确保不靠谱的代码不会被当作可信结果交付。

## 1. 何时触发方案 B
- 研究者的结构无法用方案 A 的 solid/nanotube/nanowire + 参数表达(如:锥形管腔、核壳纳米线、梯度孔隙、非圆截面)。
- 需要方案 A 未实现的打分量(如横向扩散、能谱沉积、按体素三维分布)。

## 2. 生成流程(技能驱动)
1. **复述需求 → 结构化规格**(几何/材料/源/打分,JSON)。
2. **以方案 A 源码为模板**生成新 C++(技能让 Claude 改写 `DetectorConstruction` 等,保持 PhysicsList/源/打分骨架不变)。
3. **编译**(复用方案 A 的 CMake/工具链)。
4. **回归校验(关键)**:对方案 A **能表达的**子集(如直管 TNTAs),用方案 B 生成的引擎重跑,和方案 A 的黄金 CSV 对比。
5. 仅当回归校验通过,才用方案 B 引擎跑研究者的新几何并出图;否则报告"生成代码未通过基准校验",回退或人工介入。

## 3. 以方案 A 为基准的验证方法学
- **基准数据**:方案 A 黄金测试 CSV(`data/TSC|TNTAs|ETNTAs_edd_edr.csv`)。
- **比较工具**:`analysis/compare_runs.py A_prefix B_prefix --tol 0.08`
  - 比较项:EDR 饱和值相对误差、90% 沉积深度(最优高度)相对误差、EDD 峰深度。
  - 通过判据:相对误差 < 容差(默认 8%)。返回码 0/1 便于自动化。
- **不变量自检**:方案 B 引擎也必须通过 `test_beta`(β 谱 17.4keV)与几何无重叠检查。

## 4. 护栏(可靠性)
- 方案 B 生成的代码**必须先在方案 A 能表达的几何上复现方案 A 的结果**,才允许用于新几何——这把"现场生成"的风险用"已验证基准"兜住。
- 任何未通过基准校验的生成结果,技能必须明确标注"未验证/不可信",绝不静默交付。
